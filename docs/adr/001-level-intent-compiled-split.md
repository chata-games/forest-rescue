# ADR-001: Level Intent vs Compiled Level Split

## Status

Accepted

## Context

Forest Rescue: Heartwood needs procedurally consistent campaign levels that LLM agents can author without corrupting gameplay geometry. Tower-defense PCG research shows that path shape, tower placement, and wave composition must be validated together — accepting a single random result produces unbalanced levels.

The original Forest Rescue prototype hardcodes a 7×4 lane grid and procedural waves in `game.js`. The Heartwood concept requires per-level paths, fairy rings, and scripted waves.

## Decision

Use two level formats:

1. **LevelIntent** — the only file LLM agents author. Contains creative direction, topology archetype, difficulty targets, constraints, unlocks, and wave policy. No coordinates.

2. **CompiledLevel** — generated by `tools/levelgen/compile.mjs`. Contains paths (control points, samples, arc-length tables), rings, landmarks, decorations, exact waves, and balance metrics. Never hand-edited.

CI recompiles all intents and fails if committed compiled output differs. This prevents agents from bypassing the generator by editing coordinates directly.

## Consequences

### Positive

- Reproducible levels from seed + compiler version.
- LLM creativity constrained to safe, validatable parameters.
- Automated balance scoring and simulation before merge.
- Clear audit trail via `intentHash` on compiled output.

### Negative

- Compiler must converge reliably; intents may need revision if constraints are too tight.
- Two files per level instead of one.
- Dev workflow requires `node serve.js` and compile step.

## Alternatives considered

- **Flat JSON with coordinates** — rejected; agents would edit geometry directly, breaking balance guarantees.
- **ImageGen backgrounds as level maps** — rejected; raster interpretation cannot drive collision, movement, or ring placement.
- **WFC as primary path generator** — rejected; good for local tile adjacency, poor for global TD puzzle balance.
